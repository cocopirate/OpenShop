## 1. 背景与目标

管理员服务（admin-service）是 OpenShop 后台管理体系的核心服务，负责管理员账号体系、基于 RBAC 的权限管理、商家审核及系统配置管理。其存在解决了以下问题：
- 将后台管理能力从业务服务中剥离，形成独立的权限主数据中心
- 为 auth-service 提供权限数据支撑，实现集中式权限控制
- 为平台运营人员提供商家审核、系统参数配置等管理能力

## 2. 用户角色

| 角色 | 描述 |
|------|------|
| 超级管理员 | 拥有所有权限，可管理其他管理员账号和角色 |
| 普通管理员 | 根据分配角色拥有部分后台管理权限 |
| auth-service | 内部服务，在管理员登录时调用获取权限数据 |
| admin-bff | 后台聚合层，代理前端管理页面的请求 |

## 3. 功能需求

### 3.1 核心功能

- 管理员账号管理：创建、编辑、删除、启用/禁用管理员账号
- 角色管理：创建、编辑、删除角色，维护角色与权限的映射关系
- 权限管理：定义系统权限项，支持按模块分组
- 管理员角色分配：为管理员分配一个或多个角色
- 商家审核：审核商家入驻申请，支持通过、拒绝、暂停操作
- 系统配置管理：维护平台级别的系统配置参数
- 内部权限查询接口：供 auth-service 在管理员登录时拉取该账号的完整权限列表

### 3.2 功能详情

#### 管理员账号管理

- 输入：用户名、密码、邮箱、角色列表
- 处理：创建账号记录，对密码进行 bcrypt hash，记录审计日志
- 输出：管理员账号详情（不含密码）
- 边界：用户名/邮箱唯一；超级管理员账号不可删除；删除操作为软删除

#### 角色与权限管理（RBAC）

- 角色（Role）：命名的权限集合，如"订单管理员"、"客服"
- 权限（Permission）：具体操作的访问控制项，如 `order:read`、`merchant:approve`
- 角色-权限关联：多对多关系，通过关联表维护
- 输入：角色名称、权限 code 列表
- 处理：创建角色，建立角色与权限的关联记录
- 输出：角色详情（含权限列表）
- 边界：权限 code 全局唯一；删除角色前需检查是否有管理员在用

#### 商家审核

- 输入：商家 ID、审核动作（approve/reject/suspend）、审核意见
- 处理：更新商家状态（调用 merchant-service 或共享数据库）→ 记录审核日志 → 触发通知事件
- 输出：审核结果
- 边界：只有处于 `pending` 状态的商家可以被审核通过或拒绝；已通过的商家可以被暂停

#### 内部权限查询接口

- 输入：管理员 ID
- 处理：查询该管理员的角色列表 → 展开所有角色的权限 → 去重合并
- 输出：权限 code 列表
- 该接口为内部接口，仅允许 auth-service 调用（通过内部网络策略限制）

## 4. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 权限查询接口 P99 < 100ms（数据量小，建议增加缓存） |
| 安全 | 所有管理接口需要 JWT 鉴权 + 权限校验；内部接口仅允许内网访问 |
| 审计 | 所有账号、角色、权限变更操作需记录操作人、时间、变更内容 |
| 可用性 | 99.9% 月可用性；admin-service 不可用时管理员登录降级，但不影响已登录用户的业务操作 |

## 5. 约束与依赖

| 依赖 | 说明 |
|------|------|
| PostgreSQL | 存储管理员账号、角色、权限、关联关系 |
| auth-service | 管理员登录时 auth-service 调用本服务获取权限 |
| merchant-service | 商家审核时需更新商家状态 |

主要约束：
- 权限数据不做跨服务缓存，auth-service 每次登录时实时拉取
- 端口 8012（参考架构文档）
- RBAC 设计详见 [RBAC 设计文档](../../05-services/admin-service/rbac.md)，数据模型详见 [数据模型文档](../../05-services/admin-service/schema.md)
