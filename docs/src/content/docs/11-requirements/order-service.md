## 1. 背景与目标

订单服务（order-service，:8005）负责订单数据的 CRUD 与状态全生命周期管理，是电商平台交易链路的核心持久化服务。其存在解决了以下问题：
- 提供标准化的订单创建、查询、更新接口，作为订单数据的唯一权威来源
- 支持订单状态流转，覆盖从创建到完成/取消的完整生命周期
- 通过事件发布驱动下游服务（支付、库存、物流、通知）响应订单状态变化
- 与 order-orchestration 协作，由编排服务通过 Saga 模式调用本服务完成原子性订单流程

## 2. 用户角色

| 角色 | 描述 |
|------|------|
| order-orchestration | 内部编排服务，通过 Saga 模式调用 order-service 创建/更新订单 |
| 消费者 | 查询自己的订单列表和订单详情；发起取消申请 |
| 商家 | 查询店铺订单；进行发货操作 |
| 管理员 | 查询全量订单；处理纠纷和异常订单 |
| 支付服务 | 支付成功后通知 order-service 更新订单状态为已支付 |

## 3. 功能需求

### 3.1 核心功能

- 创建订单：由 order-orchestration 调用，创建订单记录（状态 `created`）
- 订单查询：按消费者/商家/管理员维度查询订单列表和详情
- 订单状态更新：根据业务事件推进订单状态流转
- 发货管理：商家填写物流单号，将订单状态更新为 `shipped`
- 取消订单：消费者或系统发起取消，记录取消原因
- 完成订单：消费者确认收货或超时自动确认，状态变更为 `completed`
- 事件发布：在订单状态变更时发布对应领域事件

### 3.2 功能详情

#### 订单状态流转

```
created → paid        （支付成功，由支付服务回调触发）
created → cancelled   （支付超时或用户取消）
paid → shipped        （商家发货）
paid → refunding      （申请退款）
shipped → completed   （消费者确认收货或超时自动完成）
shipped → refunding   （签收后申请退款）
refunding → refunded  （退款成功）
```

#### 创建订单

- 调用方：order-orchestration（内部接口）
- 输入：消费者 ID、商家 ID、店铺 ID、订单行（商品 ID、SKU ID、数量、单价）、收货地址快照、优惠信息
- 处理：生成唯一订单号 → 计算订单金额 → 持久化订单及订单行 → 发布 `order.created` 事件
- 输出：订单 ID、订单号、订单状态、创建时间
- 幂等：通过请求中携带的 `idempotency_key`（由 order-orchestration 生成）保证幂等

#### 订单查询

- 消费者维度：返回消费者名下所有订单，支持按状态过滤、分页
- 商家维度：返回店铺名下所有订单，支持按状态/时间过滤、分页
- 管理员维度：全量查询，支持多条件过滤
- 订单详情：返回完整订单信息，包括订单行、收货地址、状态变更历史

#### 发货管理

- 输入：订单 ID、物流公司 code、物流单号
- 处理：校验订单状态为 `paid` → 更新状态为 `shipped` → 记录物流信息 → 发布 `order.shipped` 事件
- 边界：只有订单归属商家才能操作发货

#### 事件发布

| 事件名 | 触发时机 |
|--------|----------|
| `order.created` | 订单创建成功 |
| `order.paid` | 支付成功，状态变为 paid |
| `order.shipped` | 商家发货，状态变为 shipped |
| `order.cancelled` | 订单取消 |
| `order.completed` | 订单完成（确认收货） |

## 4. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 创建订单 P99 < 500ms；查询接口 P99 < 200ms |
| 幂等性 | 创建订单接口通过 `idempotency_key` 保证幂等；状态更新通过乐观锁防并发冲突 |
| 一致性 | 订单创建与事件发布通过本地消息表或事务性发件箱（Outbox）保证最终一致性 |
| 安全 | 消费者只能查询/操作自己的订单；商家只能操作归属本店铺的订单 |
| 可用性 | 99.9% 月可用性；订单数据不可丢失 |

## 5. 约束与依赖

| 依赖 | 说明 |
|------|------|
| PostgreSQL | 持久化订单数据 |
| RabbitMQ | 发布订单领域事件 |
| order-orchestration（:8100） | 上游调用方，通过 Saga 编排调用 order-service 创建订单 |
| consumer-service | 下单时校验消费者信息（由 order-orchestration 负责，非 order-service 直接调用） |

主要约束：
- order-service 是订单数据的唯一持久化层，其他服务不直接读写订单表
- 端口 8005
- 订单号格式由本服务定义，需全局唯一且有业务含义（如含日期前缀）
