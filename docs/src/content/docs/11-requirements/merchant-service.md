## 1. 背景与目标

商家服务（merchant-service，:8002）负责商家入驻流程与店铺全生命周期管理，是平台供给侧的核心服务。其存在解决了以下问题：
- 统一管理商家从申请到审核再到上线的完整入驻流程
- 维护商家基本信息、营业执照等资质数据，支撑合规运营
- 管理商家下属店铺的创建与状态，为商品服务提供店铺归属数据
- 管理商家状态流转，支撑运营侧对商家的审核与管控

## 2. 用户角色

| 角色 | 描述 |
|------|------|
| 商家申请人 | 提交入驻申请，完善资质信息 |
| 管理员 | 通过 admin-service 审核商家申请，进行暂停/恢复操作 |
| 商家（已认证） | 管理店铺信息、员工账号 |
| product-service | 查询店铺信息，校验商品归属 |
| auth-service | 验证商家账号登录凭证 |

## 3. 功能需求

### 3.1 核心功能

- 商家入驻申请：商家提交入驻申请，填写基本信息和资质材料
- 商家审核：管理员审核申请，支持通过/拒绝/暂停/恢复操作
- 商家信息管理：查询、更新商家基本信息和联系方式
- 店铺管理：创建、编辑、查询商家名下的店铺
- 商家状态管理：维护商家状态流转（pending → approved/rejected；approved ↔ suspended）
- 员工账号管理：商家管理名下子账号/员工账号的权限

### 3.2 功能详情

#### 商家入驻申请

- 输入：商家名称、联系人、手机号、邮箱、营业执照号、营业执照图片 URL、经营类目
- 处理：创建商家记录（状态 `pending`）→ 通知管理员待审核
- 输出：商家 ID、申请状态、提交时间
- 边界：同一手机号/营业执照号已存在待审核或已通过的申请时返回 409

#### 商家状态流转

```
pending → approved   （管理员审核通过）
pending → rejected   （管理员审核拒绝，可重新提交）
approved → suspended （管理员暂停）
suspended → approved （管理员恢复）
```

- 状态变更需记录操作人、时间、原因
- 状态变更后发布对应事件（`merchant.approved`、`merchant.rejected`、`merchant.suspended`）

#### 商家信息管理

- 支持商家自行更新：联系人、手机号、邮箱、店铺简介
- 营业执照等资质信息变更需重新进入审核流程
- 输入：商家 ID（来自 JWT）、更新字段
- 输出：更新后的商家信息

#### 店铺管理

- 创建店铺：商家（状态 `approved`）可创建店铺，一个商家可拥有多个店铺
- 输入：店铺名称、店铺类目、描述、logo URL
- 处理：创建店铺记录，关联商家 ID
- 输出：店铺 ID、创建时间
- 边界：商家状态非 `approved` 时不允许创建店铺；单商家最多 10 个店铺

#### 员工账号管理

- 商家可创建子账号，分配特定权限（如：仅管理订单、仅管理商品）
- 输入：员工手机号、姓名、角色（owner/staff）
- 输出：员工账号 ID
- 边界：单商家最多创建 50 个员工账号

## 4. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 查询接口 P99 < 200ms |
| 安全 | 商家只能查询和修改自己的数据；营业执照图片通过 OSS 签名 URL 访问 |
| 审计 | 商家状态变更、审核操作需完整审计日志 |
| 可用性 | 99.9% 月可用性 |
| 幂等性 | 入驻申请使用营业执照号唯一约束 |

## 5. 约束与依赖

| 依赖 | 说明 |
|------|------|
| PostgreSQL | 存储商家、店铺、员工账号数据 |
| admin-service | 商家审核操作由 admin-service 发起调用 |
| auth-service | 商家账号登录验证，auth-service 会查询商家账号状态 |
| RabbitMQ | 发布商家状态变更事件 |
| OSS | 存储营业执照等资质图片 |

主要约束：
- 端口 8002
- 商家状态为 `approved` 才能进行商品上架、接单等操作
- 商家 ID 作为多个下游服务（product-service、order-service）的租户标识
