# 服务拆分指南

## 服务拆分原则

### 1. 按业务域拆分（DDD 限界上下文）

每个微服务对应一个独立的业务域（限界上下文），域内数据高内聚，域间通过 API 或事件低耦合：

- user-service：用户域（客户 + 管理员）
- merchant-service：商家域
- product-service：商品域
- order-service：订单域
- inventory-service：库存域
- promotion-service：促销域
- aftersale-service：售后域

### 2. 能力服务独立

与业务无关的技术能力（短信、隐私号）抽取为独立能力服务，避免业务服务直接依赖外部供应商：

- sms-service：短信能力
- virtual-number-service：隐私号能力

### 3. 聚合层分离

BFF（Backend For Frontend）聚合多个服务的数据，避免客户端直接请求多个微服务：

- app-bff：聚合移动端所需数据
- admin-bff：聚合管理后台所需数据

## 何时应该拆分服务？

✅ 适合拆分的场景：
- 业务域边界清晰，团队可独立开发维护
- 不同服务有不同的扩缩容需求（如商品查询 QPS 远高于订单创建）
- 不同服务有不同的技术栈需求
- 需要独立部署和发布节奏

❌ 不适合拆分的场景：
- 服务过小，频繁跨服务调用带来的网络开销超过拆分收益
- 数据强耦合，无法避免分布式事务
- 团队规模小，维护多服务的运维成本过高

## 服务间通信

| 通信方式 | 适用场景 | 示例 |
|---------|---------|------|
| 同步 HTTP | 需要立即获取结果的查询/命令 | 结账时校验优惠券 |
| 异步消息 | 不需要立即响应的事件通知 | 订单创建后发送通知 |
| 共享库 | 纯代码复用（无网络调用） | DTO、工具函数 |

## 避免的反模式

❌ **共享数据库**：禁止多个服务直接读写同一个数据库表  
❌ **同步调用链过深**：A→B→C→D 超过 3 层同步调用，考虑改为异步  
❌ **服务过细化**：每个函数一个服务（纳米服务反模式）
